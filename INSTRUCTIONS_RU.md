# Инструкция по использованию (RU)

## Что это

Репозиторий с тремя экспериментами и прототипом pipeline для цифрового
инструмента мотивации ППС на основе эмоционального интеллекта. Все данные —
открытые международные датасеты (GoEmotions, ISEAR) плюс 75 авторских
сценариев-«виньеток» о типичных ситуациях ППС (`data/vignettes/`) — это
**сценарный stress-test, не датасет реальных данных ППС**, так и
формулируйте в статье. Никакие данные реальных преподавателей не
собирались и не выдаются за собранные.

## Установка

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Если `pip install` ругается на системные ограничения, добавьте флаг
`--break-system-packages`.

## Быстрый старт

```bash
# быстрый тестовый прогон (~10 секунд, маленькая подвыборка)
python run_all.py --quick

# полный прогон на всех данных (рекомендуется для финальных результатов)
python run_all.py

# дополнительно: попытаться обучить трансформер в Эксперименте 1
# (нужен интернет-доступ к Hugging Face Hub — лучше запускать на обычном
# компьютере/сервере, не в изолированной песочнице)
python run_all.py --use_transformer

# дополнительно: четвёртый классификатор через LLM (few-shot, не keyword-
# matching) — нужен ANTHROPIC_API_KEY, реально тратит деньги на вызовы API
# (но кэшируется в results/.llm_cache.json, повторные запуски бесплатны)
python run_all.py --use_llm

# параметры для надёжности (стратифицированная выборка по классам, больше
# few-shot примеров, параллельные вызовы):
python run_all.py --use_llm --llm_sample_size 150 --llm_examples_per_label 5 --llm_workers 8
```

### Сколько это будет стоить и сколько займёт времени (полный масштаб)

При дефолтных настройках (150 примеров на класс в Exp.1 — стратифицированно,
т.е. редкие классы типа disgust/fear в GoEmotions берутся целиком, если их
меньше 150; 5 few-shot примеров на класс; 75 виньеток в Exp.2) — это
**≈1730 вызовов API, ≈1.7 млн входных токенов**:

| модель | ориентировочная стоимость | время (8 параллельных потоков) |
|---|:---:|:---:|
| `claude-sonnet-4-6` (по умолчанию) | ≈ $5 | ≈ 6-7 минут |
| `claude-haiku-4-5-20251001` (через `ANTHROPIC_MODEL`) | ≈ $1.5-2 | ≈ 4-5 минут |

Это оценка по токенам, не гарантия — реальную сумму смотрите в консоли.
Можно уменьшить `--llm_sample_size` или `--llm_examples_per_label`, если
хочется дешевле. Если прогон прервётся (рейт-лимит, сеть) — результаты
кэшируются, повторный запуск досчитает только то, что не успело пройти.

### Сколько стоит новый 4-й уровень сообщений (`full_context`)

Это отдельная, гораздо более дешёвая статья расходов — **классификацию
(Exp.1/Exp.2) пересчитывать заново НЕ нужно**, если у вас уже есть
сохранённые `results/exp1_metrics.json`/`exp2_predictions.csv` с
`llm_fewshot`. Новый уровень добавляет всего 75 вызовов (только oracle)
или 150 (oracle + end-to-end) — короткие промпты:

| | вызовов | токенов | стоимость (Sonnet) | время (последовательно) |
|---|:---:|:---:|:---:|:---:|
| только oracle | 75 | ~0.02M | ≈ $0.10 | ≈ 2 мин |
| oracle + end-to-end | 150 | ~0.04M | ≈ $0.25 | ≈ 4-5 мин |

Команда:
```bash
python experiments/exp3_intervention_quality.py
# или явно пропустить дорогой уровень:
python experiments/exp3_intervention_quality.py --skip_full_context
```

При первом запуске скрипты сами скачают GoEmotions и ISEAR (~4 МБ, GitHub)
в `data/raw/` и закэшируют — повторные запуски не требуют интернета.

## Что лежит в `results/` после запуска

- `exp1_metrics.json` — точность/F1 классификаторов эмоций, in-domain и
  cross-dataset, плюс `out_of_label_space_rate` (см. ниже, почему это
  важно), плюс `tfidf_logreg_5class` — честная версия TF-IDF.
- `results/figures/*.png` — confusion matrix для каждой модели **включая
  LLM** (раньше для LLM картинок не было — это было упущение, теперь
  исправлено), плюс `domain_shift_ladder.png` (сводный график точности по
  трём точкам domain shift для всех моделей — главная иллюстрация для
  статьи) и `exp3_generic_vs_oracle.png` / `exp3_generic_vs_e2e.png`
  (парный график «обычное сообщение → персонализированное» по каждой
  виньетке).
- `exp2_metrics.json`, `exp2_predictions.csv` — точность на виньетках, по
  каждой эмоции и категории стрессора, для всех моделей включая честный
  `tfidf_logreg_5class`.
- `exp3_outputs.csv`, `exp3_summary.json` — теперь **4 уровня персонализации**
  (generic / emotion_only / need_only / full_context) в двух режимах
  (oracle/end-to-end), со staircase-сравнениями между соседними уровнями —
  это и отвечает на вопрос «что именно добавляет ценность: детекция эмоции,
  привязка к SDT-потребности или полный контекст ситуации».
- `results/figures/exp3_ablation_oracle.png` (и `_end_to_end.png`, если
  Exp.2 запускался с `--use_llm`) — столбчатая диаграмма по всем 4 уровням.
- `exp3_rubric_blind.csv` — **слепая** анкета на стратифицированную
  подвыборку (по умолчанию 25 виньеток, `--rubric_sample_size`), 3-4 слота
  (A/B/C/D, в зависимости от того, доступен ли full_context) × 4 шкалы:
  relevance, specificity, sdt_alignment, motivational_usefulness.
- `exp3_rubric_key.csv` — **не показывайте экспертам** — ключ "какой слот
  какой уровень", нужен только вам для анализа.
- `data/vignettes/vignette_validation_template.csv` — отдельная анкета:
  не для оценки сообщений, а для проверки самих 75 виньеток (правдоподобна
  ли ситуация, верно ли подобрана эмоция) — тоже на нескольких экспертов.

После того как эксперты заполнят `exp3_rubric_blind.csv` (каждый — свою
копию), прогоните:
```bash
python experiments/exp3b_rubric_analysis.py \
    --rubrics results/filled_rater1.csv results/filled_rater2.csv \
    --key results/exp3_rubric_key.csv
```
Скрипт сам расшифрует A/B обратно в generic/personalized, посчитает
парные t-тесты по каждому критерию (релевантность/специфичность/
уместность) и межэкспертное согласие.

## Если хотите расширенный/полный вариант

1. Запустите `python run_all.py` (без `--quick`) — это уже полный прогон
   на всём датасете GoEmotions (36 тыс. строк), занимает секунды.
2. Если нужен трансформер (BERT/RoBERTa) — запускайте `--use_transformer`
   на машине с обычным интернетом (не в этой песочнице): скрипт сам
   скачает предобученную модель с Hugging Face и обучит её.
3. Если хотите подключить реальную генерацию сообщений через LLM (а не
   шаблоны) — поставьте переменную окружения `ANTHROPIC_API_KEY` и
   запустите `python experiments/exp3_intervention_quality.py --mode llm`.
4. Если захотите добавить свои виньетки — просто допишите строки в
   `data/vignettes/faculty_vignettes.csv` (колонки: id, expected_emotion,
   sdt_need, context_tag, text), используя только эмоции anger / disgust /
   fear / joy / sadness — это единственные 5 классов, общие для обоих
   датасетов.

## Структура папок (кратко)

- `src/` — вся логика (загрузка данных, модели, маппинг на SDT, генератор
  сообщений, метрики). Английские короткие комментарии в коде.
- `experiments/` — три отдельных запускаемых скрипта, по одному на
  эксперимент.
- `data/vignettes/` — единственное, что мы «придумали» сами (виньетки и
  описания потребностей) — остальное скачивается.
- `results/` — куда сохраняются все цифры/графики/CSV для статьи.

## Что можно сразу взять в статью

- **Главный график** — `domain_shift_ladder.png`: точность по трём точкам
  (GoEmotions → ISEAR → виньетки) для всех моделей сразу.
- **Важная поправка (раньше было неточно сформулировано)**: исходное
  сравнение было нечестным — TF-IDF обучался на 7 классах и мог
  предсказать "neutral"/"surprise" на ISEAR/виньетках, где этих классов
  физически нет. Проверил: **46.5% предсказаний TF-IDF на ISEAR и 49.3% на
  виньетках уходили в эти "невозможные" классы**. Честная версия
  (`tfidf_logreg_5class`, обучена только на 5 общих классах) показывает,
  что supervised TF-IDF на самом деле обобщается **не хуже, а немного
  лучше**, чем словарный метод (0.381 vs 0.326 на ISEAR; 0.387 vs 0.360 на
  виньетках) — старый вывод "словарь обобщается лучше" был отчасти
  артефактом, не реальным эффектом.
- **LLM few-shot подтверждённо лучше всех** на обоих out-of-domain тестах
  (0.785 / 0.893) — но это уже не гипотеза, а измеренный результат с
  надёжной (стратифицированной) выборкой.
- Статистически значимое превосходство персонализированных сообщений
  (p < 0.0001) над generic — но в статье обязательно указывайте: это
  TF-IDF similarity, слабая авто-метрика, смещённая в пользу
  персонализированных шаблонов по построению. Ведущий результат для статьи
  должен быть из слепой экспертной анкеты (`exp3_rubric_blind.csv`), а не
  из этой авто-метрики одной.

## Разбор внешней критики (4 пункта) — что поправил, что осталось

Критика была дельная, по существу. Вот что сделал:

1. **"TF-IDF нечестно сравнивается"** — подтвердил эмпирически (см. выше),
   исправил: добавил `tfidf_logreg_5class`, добавил метрику
   `out_of_label_space_rate` в каждый отчёт автоматически.
2. **"Exp.3 не end-to-end, это upper bound"** — согласен, добавил второй
   режим: predicted emotion (из `llm_fewshot`) → SDT need → сообщение,
   плюс `need_match_rate` (как часто неверно предсказанная эмоция всё
   равно попадает в правильную потребность — например, anger и disgust
   обе ведут к autonomy, так что не всякая ошибка классификации портит
   сообщение).
3. **"TF-IDF similarity — слабая метрика, нужна экспертная оценка"** —
   согласен полностью. Переделал rubric на **слепую** (A/B, без подписи
   generic/personalized) и написал скрипт анализа
   (`exp3b_rubric_analysis.py`). Сами оценки экспертов — это то, что я
   физически не могу сделать за вас; шаблон готов, дело за реальными
   3–5 оценщиками.
4. **"Виньетки — не датасет ППС"** — согласен, везде в коде/докстрингах
   теперь явно "researcher-authored scenario set / vignette-based stress
   test", не "dataset". Добавил отдельную анкету для экспертной проверки
   самих виньеток (`vignette_validation_template.csv`). Реальные 30-50
   анонимных фраз от настоящих ППС — намеренно НЕ стал придумывать/подделывать,
   это и есть тот самый следующий, отдельный эмпирический этап.
5. **Технический пункт про офлайн-запуск без интернета** — не стал делать
   полноценный fallback-датасет (риск лицензионных вопросов при
   коммите чужих данных + лишняя сложность), но добавил в README.md
   (раздел 6) точные прямые ссылки на все 5 файлов с точными именами —
   рецензент может скачать руками на любой другой машине и положить в
   `data/raw/`, дальше всё работает офлайн.

## Второй раунд критики (этот чат) — что сделал

1. **"Разделить oracle/end-to-end"** — это уже было сделано в прошлый раз
   (см. выше, п.2). Если рецензент видит старую версию README/кода —
   просто перешлите обновлённый архив.
2. **"Сообщения слишком общие, нужен полный контекст (текст виньетки +
   эмоция + SDT + категория стрессора)"** — согласен, добавил 4-й уровень
   `full_context`: LLM получает реальный текст ситуации + категорию
   стрессора (7 категорий + positive_engagement — добавил колонку
   `stressor_category` в `faculty_vignettes.csv`) и пишет сообщение,
   ссылающееся на конкретную деталь, а не общую формулировку.
3. **"Минимум 3-5 экспертов, blind, 4 шкалы (relevance/specificity/
   sdt_alignment/motivational_usefulness), главная метрика — экспертная,
   не TF-IDF"** — согласен, переделал rubric именно на эти 4 шкалы,
   TF-IDF теперь явно подписан как "secondary automatic proxy" везде в
   README. Сбор реальных оценок экспертов — всё ещё ваша часть работы,
   шаблон полностью готов.
4. **"4 варианта сравнения вместо 2"** — реализовал ровно как предложено:
   generic / emotion_only / need_only / full_context, со staircase-
   сравнением каждого уровня с предыдущим. Уже на oracle-данных (без
   API) видно интересный результат: переход generic→emotion_only
   незначим (p=0.27) — то есть просто назвать эмоцию не помогает; весь
   эффект — на шаге need_only (p<0.0001), то есть именно привязка к
   SDT-теории даёт ценность, а не сама детекция эмоции. Это сильнее и
   точнее, чем старое "personalized лучше generic".

## Если что-то не запускается

- Ошибка скачивания GoEmotions/ISEAR → нет доступа к
  `raw.githubusercontent.com`. Скачайте файлы вручную по ссылкам в
  `src/data_loader.py` и положите их в `data/raw/` с нужными именами.
- Ошибка про `transformers`/`torch` → это нормально без `--use_transformer`,
  эти пакеты не обязательны для основных трёх экспериментов.
- Ошибка про `ANTHROPIC_API_KEY` → тоже нормально, без неё `exp3` просто
  использует офлайн-шаблоны (режим по умолчанию), а `--use_llm` в Exp.1/2
  просто пропускается с понятным сообщением.
- Если видите много сообщений про повтор запроса (rate limit, HTTP 429) →
  это нормально, скрипт сам делает повторные попытки с задержкой. Если
  повторов слишком много — уменьшите `--llm_workers` (например, до 3-4).

## "А можно точность 80%+ другой моделью?" (см. README.md, раздел 9)

Коротко: не через смену архитектуры (pyKAN и т.п. тут не помогут — это
архитектуры для гладких низкоразмерных функций, а не для разреженного
текста; узкое место — несовпадение домена обучения и домена применения,
а не «слабость» модели). Реально может помочь: (1) LLM few-shot
(`--use_llm`, уже встроен, нужен только ваш `ANTHROPIC_API_KEY`), (2)
трансформер (`--use_transformer`, частично поможет in-domain), (3) в
конечном счёте — реальные данные ППС, которых сейчас просто не существует.
Подробное объяснение с цифрами — в README.md, раздел 9.
